# 智能数据 ETL —— 产品需求文档（PRD）

> 版本：v1.0 | 更新日期：2026-02-26

---

## 一、产品定位

一句话描述：**通过自然语言对话，将分散的业务基表加工为可直接用于"问数"的宽表/主题表，替代人工编写复杂 SQL。**

### 核心问题

| 痛点 | 说明 |
|------|------|
| 基表分散 | 问数所需字段散落在数十张甚至上百张表中 |
| 加工逻辑复杂 | 同一字段在不同表中含义、口径不同，需要 CASE WHEN / JOIN / 窗口函数等复杂 SQL |
| 维护成本高 | 业务变化后 SQL 难以追溯和修改 |
| 质量难保障 | 缺少系统化的数据校验机制 |

---

## 二、用户角色

| 角色 | 说明 |
|------|------|
| 数据分析师 | 主要用户，熟悉业务口径但不擅长复杂 SQL |
| 数据工程师 | 审核/微调生成的 ETL 逻辑 |

## 四、Step 0 — 选择连接器（前置条件，非对话态）

### 4.1 功能说明

进入对话前，用户必须先选择一个已配置的数据库连接器。**此步骤在对话界面之外完成**，属于前置门槛。

### 4.2 交互设计

```
┌──────────────────────────────────────────────┐
│         选择数据源连接器                       │
│                                              │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐   │
│  │ MySQL    │  │ PostgreSQL│  │ ClickHouse│  │
│  │ 业务主库  │  │ 数仓      │  │ OLAP库    │  │
│  │ ✓ 已连接  │  │ ✓ 已连接  │  │ ✗ 未配置  │  │
│  └──────────┘  └──────────┘  └──────────┘   │
│                                              │
│  + 新建连接器                                 │
│                                              │
│              ┌──────────────┐                │
│              │   开始加工 ▶  │                │
│              └──────────────┘                │
└──────────────────────────────────────────────┘
```

### 4.3 连接器信息

| 字段 | 说明 |
|------|------|
| 连接器名称 | 用户自定义，如"业务主库" |
| 数据库类型 | MySQL / PostgreSQL / ClickHouse / Hive / StarRocks 等 |
| 连接地址 | host:port |
| 认证信息 | 用户名/密码，加密存储 |
| 连接状态 | 实时检测，不可选未连通的连接器 |

### 4.4 约束

- 未选择连接器时，"开始加工"按钮置灰不可点击
- 未连通的连接器不可选择，提示"连接失败，请检查配置"
- 一次对话会话绑定一个连接器，中途不可切换（需新建会话）

## 五、Step 1 — 选择源表（引导式对话）

### 5.1 功能说明

用户进入对话后，系统**主动引导**用户选定本次加工涉及的源表。不是让用户自由描述，而是通过结构化交互 + 自然语言辅助的方式完成。

### 5.2 对话流程

```

└───────────────────────────────────────────────────────────────────────────┘

┌─ 用户 ────────────────────────────────────────────────────────────────────┐
│ 我需要做一张用户消费行为分析的宽表，需要用户基础信息、订单数据、                  │
│ 商品信息和退款数据                                                         │
└───────────────────────────────────────────────────────────────────────────┘

┌─ 系统 ─────────────────────────────────────────────────────────────────────┐
│ 根据你的描述，我推荐以下源表：                                                │
│                                                                           │
│ ☑ user_info          用户基础信息表   (23列, 约150w行)                       │
```

### 5.3 约束与限制

| 规则 | 说明 |
|------|------|
| 源表上限 | 单次加工最多选择 **15 张**源表，超出提示拆分任务 |
| 最少选择 | 至少选择 **2 张**表（单表无需 ETL） |
| 表元数据缓存 | 选定后系统自动拉取并缓存表结构（列名、类型、注释、采样数据） |
| 不支持跨库 | 所有源表必须来自同一连接器 |

## 六、Step 2 — 定义目标表（核心对话阶段）

### 6.1 功能说明

这是产品的核心环节。用户通过自然语言描述需要哪些字段、每个字段的加工口径，系统将其翻译为 SQL 逻辑。

**关键设计原则：不是自由聊天，而是"逐字段定义"的结构化对话。**

### 6.2 对话模式：字段工作台

系统将对话分为两种模式，用户可随时切换：

#### 模式 A：逐字段引导模式（推荐新手）

系统逐个引导用户定义字段，每轮对话聚焦一个字段。

```
┌─ 系统 ─────────────────────────────────────────────────────────────────────┐
│ 已选定 6 张源表，共 153 个字段可用。                                          │
│                                                                           │
│ 现在开始定义目标表「用户消费行为宽表」的字段。                                   │
│                                                                           │
│ 📋 当前字段清单（0/? 已定义）                                                │
│ ──────────────────────────                                                │
│ （空）                                                                     │
│                                                                           │
│ 请描述你需要的第一个字段，例如：                                               │
│  • "用户ID，取 user_info 表的 user_id"                                      │
│  • "最近一次下单时间"                                                       │
│  • "累计消费金额，排除已退款订单"                                              │
└───────────────────────────────────────────────────────────────────────────┘

┌─ 用户 ────────────────────────────────────────────────────────────────────┐
│ 累计消费金额，需要排除全额退款的订单                                           │
└───────────────────────────────────────────────────────────────────────────┘

┌─ 系统 ─────────────────────────────────────────────────────────────────────┐
│ 已理解，为你生成字段定义：                                                    │
│                                                                           │
│ ┌─ 字段卡片 ──────────────────────────────────────────────────────────┐    │
│ │ 字段名：total_consumption_amount                                   │    │
│ │ 中文名：累计消费金额                                                 │    │
│ │ 类型：  DECIMAL(18,2)                                              │    │
│ │ 口径：  订单实付金额汇总，排除 refund_type='全额退款' 的订单            │    │
│ │                                                                    │    │
│ │ SQL 逻辑：                                                         │    │
│ │ ┌────────────────────────────────────────────────────────────────┐ │    │
│ │ │ SUM(CASE                                                      │ │    │
│ │ │   WHEN o.order_id NOT IN (                                    │ │    │
│ │ │     SELECT order_id FROM refund_records                       │ │    │
│ │ │     WHERE refund_type = '全额退款'                             │ │    │
│ │ │   )                                                           │ │    │

│ │ │ END)                                                          │ │    │
│ │ └────────────────────────────────────────────────────────────────┘ │    │
│ │                                                                    │    │
│ │  ✅ 确认    ✏️ 修改    🗑️ 删除                                     │    │
│ └────────────────────────────────────────────────────────────────────┘    │
│                                                                           │
│ 请确认或修改，然后继续定义下一个字段。                                         │
└───────────────────────────────────────────────────────────────────────────┘
```

### 6.3 对话行为约束

以下规则限定对话边界，防止用户偏离加工任务：

| 约束 | 说明 | 触发响应 |
|------|------|---------|
| **意图识别** | 系统只响应与字段定义相关的意图 | 无关问题回复"当前处于数据加工模式，请描述你需要的字段或加工逻辑" |
| **禁止直写 SQL** | 不允许用户直接输入完整 SQL 作为字段定义 | "请用自然语言描述你的需求，我来生成 SQL" |
| **歧义必追问** | 当自然语言存在多种 SQL 理解时，系统必须追问 | 列出可能的理解，让用户选择（如上面"用户等级"的例子） |
| **字段粒度** | 每次对话回合处理 1~10 个字段 | 超过 10 个字段提示分批处理 |
| **上下文锁定** | 对话始终围绕当前目标表，不可跳转做其他表 | "当前正在定义「XX宽表」，请先完成当前表的定义" |
| **引用校验** | 用户提到的列名必须在源表中存在 | "在已选源表中未找到字段 xxx，你是否指的是 yyy？" |

### 6.4 系统智能能力

| 能力 | 说明 |
|------|------|
| **字段推荐** | 基于已选源表和用户已定义字段，推荐可能还需要的字段 |
| **口径消歧** | 同一业务含义在多表中存在时，主动询问取数口径 |
| **JOIN 推断** | 自动推断表间关联关系，用户无需指定 JOIN 条件 |
| **类型推断** | 根据加工逻辑自动推断目标字段类型 |
| **命名规范** | 自动生成符合命名规范的英文字段名 |

## 七、Step 3 — 校验确认

### 7.1 功能说明

在生成最终 SQL 前，系统自动执行多层校验，确保加工逻辑正确、数据质量可控。

### 7.2 校验层级

#### 第一层：逻辑校验（自动执行）

| 校验项 | 说明 | 示例 |
|--------|------|------|
| JOIN 合理性 | 检查表间关联是否存在笛卡尔积风险 | "orders JOIN order_items 可能导致行数膨胀，粒度从订单变为订单明细" |
| 字段类型兼容 | 检查加工逻辑中的类型是否匹配 | "user_id 在 user_info 中为 BIGINT，在 orders 中为 VARCHAR，需要类型转换" |
| NULL 处理 | LEFT JOIN 场景下未处理 NULL | "has_refund 字段在无退款记录时为 NULL，建议改为 COALESCE(..., '否')" |
| 聚合粒度 | 检查聚合字段与非聚合字段是否一致 | "top_category 使用了子查询，请确认粒度为用户级别" |

#### 第二层：数据采样校验（自动执行）

系统自动抽取 **100 条**样本数据执行生成的 SQL，展示结果。


### 7.3 校验约束

| 规则 | 说明 |
|------|------|
| 校验不可跳过 | "下一步"按钮必须在所有自动校验通过后才可点击 |
| 采样失败阻断 | 如果采样 SQL 执行失败，必须修复后才能继续 |
| 用户必须确认 | 预览数据需要用户显式点击"符合预期"才可进入下一步 |
| 修改可回溯 | 校验阶段的修改会同步回字段定义，保持一致性 |

## 八、Step 4 — 执行与发布

### 8.1 功能说明

校验通过后，用户可以选择执行方式。

### 8.2 执行选项

```
┌─ 系统 ─────────────────────────────────────────────────────────────────────┐
```

## 九、对话状态机

整个对话过程遵循严格的状态机模型，防止用户跳步或偏离流程：

```
                    ┌─────────┐
                    │  INIT   │ ← 选择连接器后进入
                    └────┬────┘
                         │ 系统展示表列表
                         ▼
                  ┌──────────────┐
              ┌──▶│ SELECT_TABLE │◀── 用户增删表
              │   └──────┬───────┘
              │          │ 用户确认源表
              │          ▼
              │   ┌──────────────┐
              │   │ DEFINE_FIELD │◀─┐
              │   └──────┬───────┘  │
              │          │          │ 用户修改字段
              │          │          │
              │          ▼          │
              │   ┌──────────────┐  │
              │   │  VALIDATE    │──┘ 校验不通过
              │   └──────┬───────┘
              │          │ 校验全部通过 + 用户确认
              │          ▼
              │   ┌──────────────┐
              └───│   PUBLISH    │ ← 可回退到任一步骤
                  └──────────────┘
```

### 状态切换规则

| 当前状态 | 允许的操作 | 不允许的操作 |
|---------|-----------|------------|
| INIT | 搜索表、选择表、描述需求 | 定义字段、校验、执行 |
| SELECT_TABLE | 增删表、确认表、描述数据范围 | 定义字段、校验、执行 |
| DEFINE_FIELD | 增删改字段、切换模式、完成定义 | 选择表（需回退）、执行 |
| VALIDATE | 确认、修改字段（回退）、查看SQL | 选择表（需回退）、执行 |
| PUBLISH | 选择执行方式、回退到任意步骤 | - |

## 十、边界场景处理

| 场景 | 处理方式 |
|------|---------|
| 用户输入与 ETL 无关的内容 | "当前处于数据加工会话中，请描述你需要的字段或加工逻辑。如需其他帮助，请新建对话。" |
| 用户要求修改连接器 | "当前会话已绑定「业务主库」连接器，如需切换请新建加工会话。" |
| 用户直接粘贴大段 SQL | "检测到你输入了 SQL 语句。本工具通过自然语言生成 SQL，请用业务语言描述你的需求。" |
| 源表数据量过大 (>1亿行) | 采样校验时提示预计执行时间，建议增加分区过滤条件 |
| 表结构变更 | 会话开始时快照表结构，会话中不感知变更；下次新建会话时刷新 |
| 对话中断后恢复 | 保存完整对话状态，用户可在 24 小时内恢复到中断前的步骤 |
| 字段名冲突 | 系统自动添加表前缀消歧，如 `user_info_name` vs `products_name` |
